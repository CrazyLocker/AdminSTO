# План исправления бага: Создание связи "одна услуга + три запчасти"

## Проблема

При создании связи "одна услуга + три запчасти" создается три отдельных записи в таблице `service_spare_parts` - по одной записи на каждую запчасть. Пользователь ожидает, что должна создаваться **одна связь с списком из трех запчастей**.

## Текущая реализация

### Структура данных

**Таблица базы данных:**
```sql
CREATE TABLE service_spare_parts (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    service_id INTEGER NOT NULL,
    spare_part_id INTEGER NOT NULL,
    quantity INTEGER DEFAULT 1,
    unit_type TEXT DEFAULT 'шт',
    active INTEGER DEFAULT 1,
    FOREIGN KEY (service_id) REFERENCES services(id),
    FOREIGN KEY (spare_part_id) REFERENCES spare_parts(id)
)
```

**Сущность `ServiceSparePart`:**
- `serviceId` - ID услуги
- `sparePartId` - ID одной запчасти  
- `quantity` - количество этой запчасти
- `unitType` - единица измерения

**Текущий алгоритм** (`SettingsView.java`, метод `showAddServiceSparePartDialog()`):
```java
// Для каждой выбранной запчасти создается отдельная запись
for (Object child : partsCheckboxes.getChildren()) {
    if (cb.isSelected()) {
        SparePart part = (SparePart) cb.getUserData();
        ServiceSparePart relation = new ServiceSparePart();
        relation.setServiceId(service.getId());
        relation.setSparePartId(part.getId()); // ← Каждая запчасть имеет свой ID
        relation.setQuantity(quantity);
        SettingsController.addServiceSparePart(relation);
    }
}
```

## Проблема

Текущая реализация создает **3 отдельные связи**:
| id | service_id | spare_part_id | quantity | unit_type |
|----|------------|---------------|----------|-----------|
| 1  | 5          | 10            | 2        | шт        |
| 2  | 5          | 15            | 1        | шт        |
| 3  | 5          | 20            | 3        | л         |

## Решение: Отдельная таблица для хранения списков

Создать новую таблицу `service_spare_parts_lists` для хранения списков запчастей, связанных с услугой.

### Новая структура таблицы

```sql
CREATE TABLE service_spare_parts_lists (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    service_id INTEGER NOT NULL,
    created_date TEXT NOT NULL,
    active INTEGER DEFAULT 1,
    FOREIGN KEY (service_id) REFERENCES services(id)
)

CREATE TABLE service_spare_parts_list_items (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    list_id INTEGER NOT NULL,
    spare_part_id INTEGER NOT NULL,
    quantity INTEGER DEFAULT 1,
    unit_type TEXT DEFAULT 'шт',
    FOREIGN KEY (list_id) REFERENCES service_spare_parts_lists(id),
    FOREIGN KEY (spare_part_id) REFERENCES spare_parts(id)
)
```

**Описание:**
- `service_spare_parts_lists` - основная запись связи "услуга + список запчастей"
- `service_spare_parts_list_items` - элементы списка (каждая запчасть с количеством)

### Новые сущности

**1. ServiceSparePartsList** (основная запись):
- `id` - ID записи
- `serviceId` - ID услуги
- `createdDate` - дата создания
- `active` - активна ли связь
- `items` - список элементов

**2. ServiceSparePartsListItem** (элемент списка):
- `id` - ID элемента
- `listId` - ID списка
- `sparePartId` - ID запчасти
- `quantity` - количество
- `unitType` - единица измерения

### Текущая структура (для сравнения)

Таблица `service_spare_parts` содержит **отдельные записи** для каждой запчасти:
| id | service_id | spare_part_id | quantity | unit_type |
|----|------------|---------------|----------|-----------|
| 1  | 5          | 10            | 2        | шт        |
| 2  | 5          | 15            | 1        | шт        |
| 3  | 5          | 20            | 3        | л         |

### Новая структура (после исправления)

При создании связи "услуга + 3 запчасти" создается:

**Одна запись в `service_spare_parts_lists`:**
| id | service_id | created_date | active |
|----|------------|--------------|--------|
| 1  | 5          | 2026-07-08   | 1      |

**Три записи в `service_spare_parts_list_items`:**
| id | list_id | spare_part_id | quantity | unit_type |
|----|---------|---------------|----------|-----------|
| 1  | 1       | 10            | 2        | шт        |
| 2  | 1       | 15            | 1        | шт        |
| 3  | 1       | 20            | 3        | л         |

**Важно:** Пользователь видит **одну связь** в интерфейсе, но внутри хранятся элементы списка.

## Рекомендуемый подход

**Создать отдельную таблицу для хранения списков запчастей:**

1. **Создать новую таблицу** `service_spare_parts_lists` - основная запись связи

## Обратная совместимость

Для сохранения данных из старой структуры используется автоматическая миграция:
- При первом подключении к БД выполняется метод `migrateServiceSparePartsToNewStructure()`
- Старые записи из `service_spare_parts` переносятся в новую структуру `service_spare_parts_lists` + `service_spare_parts_list_items`
- Старая таблица `service_spare_parts` остается для обратной совместимости - `AutoAddSparePartService` читает данные из обеих структур

### Миграция данных

При инициализации базы данных:
1. Создаются новые таблицы `service_spare_parts_lists` и `service_spare_parts_list_items`
2. Старые связи группируются по `service_id`
3. Для каждой услуги создается один список
4. Все запчасти этой услуги переносятся в элементы списка
5. Количество мигрированных записей выводится в лог

**Пример:**
Если было 3 записи:
| service_id | spare_part_id | quantity | unit_type |
|------------|---------------|----------|-----------|
| 5 | 10 | 2 | шт |
| 5 | 15 | 1 | шт |
| 5 | 20 | 3 | л |

После миграции:
- Одна запись в `service_spare_parts_lists`: service_id=5
- Три записи в `service_spare_parts_list_items`: list_id=1, spare_part_id=10/15/20

### Исправление ошибки SQLite JDBC driver:

**Баг:** `not implemented by SQLite JDBC driver` - метод `getGeneratedKeys()` не поддерживается в SQLite.

**Исправление:** Используем `last_insert_rowid()` для получения последнего вставленного ID:

```java
// Для SQLite используем last_insert_rowid()
String getIdSql = "SELECT last_insert_rowid()";
try (Statement stmt = conn.createStatement();
     ResultSet rs = stmt.executeQuery(getIdSql)) {
    if (rs.next()) {
        list.setId(rs.getInt(1));
    }
}
```

**Обновленные методы:**
- `addServiceSparePartsList()` - теперь использует `last_insert_rowid()`
- `addServiceSparePartsListItem()` - теперь использует `last_insert_rowid()`

### Исправление бага с записью:

**Баг:** После создания связи связь не сохранялась в базу данных.

**Причина:** Список `ServiceSparePartsList` не помечался как `dirty`, поэтому метод `save()` в `DataStore` не сохранял его.

**Исправление:** В `DataStore.addServiceSparePartsList()` добавлена строка `list.setDirty(true)`:
```java
public static void addServiceSparePartsList(ServiceSparePartsList list) {
    DatabaseFactory.getDatabase().addServiceSparePartsList(list);
    serviceSparePartsLists.add(list);
    
    // Помечаем список как dirty для сохранения
    list.setDirty(true);
    
    // ... остальной код
}
```

**Как это работает:**
1. Пользователь создает связь "услуга + 3 запчасти"
2. Создается объект `ServiceSparePartsList` с 3 элементами
3. Вызывается `SettingsController.addServiceSparePartsList(list)`
4. Метод сохраняет список в `DataStore` и помечает его как `dirty`
5. При следующем вызове `DataStore.save()` список сохраняется в БД
6. Вызывается `loadServiceSparePartsLists()` - таблица обновляется
