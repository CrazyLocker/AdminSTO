# KODA.md — Контекст проекта "Администратор СТО"

## Обзор проекта

**AutoService Admin** — настольное приложение для автоматизации работы администратора автосервиса (СТО). Приложение обеспечивает управление клиентами, заказами-нарядами, справочниками услуг и запчастей, а также планирование записей на ремонт.

### Назначение
- Ведение базы клиентов с информацией об автомобилях
- Создание и отслеживание заказов-нарядов на ремонт
- Управление справочниками (услуги, запчасти, склад)
- Планирование записей через календарь (день/неделя/месяц)
- Сводная панель управления (дашборд) с ключевыми метриками

### Основные технологии
| Компонент | Технология |
|-----------|------------|
| Язык | Java 17 |
| GUI | JavaFX 17.0.6 (controls, fxml) |
| База данных | SQLite (файл `autoservice.db`) |
| Сборка | Maven |
| Тестирование | JUnit 5, Mockito, AssertJ |
| Форматы данных | Gson (JSON) |

### Сущности данных

**Client** — клиент: id, имя, фамилия, телефон, модель авто, госномер, дата последнего ремонта

**WorkOrder** — заказ-наряд: id (формат `ZAK-DD/MM/YY-NNNN`), клиент, статус, список услуг, список запчастей, итоговая сумма

**Service** — услуга: название, цена, длительность, артикул

**SparePart** — запчасть: id, название, артикул, производитель, совместимые модели, закупочная/розничная цена, остаток, мин. остаток, место хранения

**Appointment** — запись: id, клиент, мастер, услуга, дата, время, статус, ссылка на заказ

### База данных (SQLite)

Файл: `autoservice.db`

Таблицы:
- `clients` — клиенты
- `services` — услуги (уникальность по name)
- `spare_parts` — запчасти (уникальность по name)
- `orders` — заказы
- `order_services` — услуги в заказе
- `order_parts` — запчасти в заказе
- `appointments` — записи на приём

Генерация ID заказа: `ZAK-DD/MM/YY-NNNN` (дата + порядковый номер).

## Сборка и запуск

### Предварительные требования
- JDK 17+
- Maven 3.6+

### Команды

| Действие | Команда |
|----------|---------|
| Сборка проекта | `mvn clean compile` |

## Правила разработки

### Стиль кодирования
- Пакет: `com.autoservice`
- Подпакеты: `controllers`, `dialogs`, `services`, `views`
- Java 17, кодировка UTF-8
- Комментарии на русском языке
- Статические методы для сервисных операций
- UI-компоненты создаются программно (без FXML)

### Практики
- Данные кэшируются в `DataStore` (в памяти), синхронизируются с БД при сохранении
- Обновление UI через контроллеры (`DictionaryController.refreshAll()`, `OrderController.refreshTable()` и т.д.)
- Автообновление дашборда при переключении на вкладку
- Валидация ввода через `Validators` (телефон, госномер)

### Тестирование
- Пакет тестов: `com.autoservice`
- Тестируемые модули: модели, валидаторы, сервисы, контроллеры, интеграционные сценарии
- Фреймворк: JUnit 5 с AssertJ и Mockito

### Структура вкладок UI
1. **Дашборд** — статистика, выручка, графики
2. **Клиенты** — CRUD, поиск
3. **Заказы** — создание, редактирование, статусы
4. **Справочники** — услуги, запчасти, склад
5. **Запись** — календарь (день/неделя/месяц)

## Частые задачи

### Добавить новую сущность
1. Создать класс модели в корне `com.autoservice`
2. Добавить методы в `Database.java` (CRUD)
3. Добавить методы в `DataStore.java` (кэш)
4. Создать контроллер в `controllers/` (если нужно обновление UI)
5. Создать диалог в `dialogs/` (если нужно взаимодействие с пользователем)
6. Добавить вью в `views/` (если нужен новый раздел UI)

### Добавить новый статус заказа
1. Обновить список статусов в `WorkOrder.java`
2. Добавить логику обработки в соответствующих контроллерах/диалогах
3. Обновить валидацию при смене статуса

### Изменить структуру БД
1. Обновить `createTables()` в `Database.java`
2. Обновить соответствующие методы CRUD
3. Обновить `DataStore.java` при необходимости
