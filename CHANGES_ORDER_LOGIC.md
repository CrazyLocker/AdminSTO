# Исправления логики работы с заказами

## Дата: 18.07.2026

### Выполненные изменения

#### 1. Исправление генерации ID заказа (SQLiteDatabase.java)

**Проблема**: 
- Метод `generateOrderId()` использовал конкатенацию SQL-строк, что небезопасно (возможность SQL-инъекций)
- Формат ID был неправильным: `ZAK-ДД-ММ-ГГ-0001` вместо `ZAK-ДД/ММ/ГГ-0001`
- Генерация номера привязывалась к году (искла только заказы с `26` в паттерне), что позволяло создавать дубликаты с разными датами

**Решение**:
- Заменена конкатенация на `PreparedStatement` для безопасности
- Изменён формат ID на `ZAK-ДД/ММ/ГГ-0001` (с косыми чертами, как в существующих данных)
- Генерация теперь использует **сквозную нумерацию**: ищется максимальный порядковый номер среди ВСЕХ заказов, независимо от даты
- Добавлено логирование для отладки генерации ID
- Добавлена проверка на неожиданный формат ID
- Улучшена обработка ошибок

**Изменения**:
```java
// Было:
String yearSuffix = String.valueOf(today.getYear()).substring(2); // "26"
String sql = "SELECT MAX(id) as max_id FROM orders WHERE id LIKE ?";
String likePattern = "ZAK-%" + yearSuffix + "-%"; // Ищет только заказы за текущий год
String date = today.format(DateTimeFormatter.ofPattern("dd-MM-yy")); // С дефисами
String orderId = String.format("ZAK-%s-%04d", date, newNumber); // ZAK-ДД-ММ-ГГ-0001

// Стало:
String sql = "SELECT MAX(id) as max_id FROM orders WHERE id LIKE 'ZAK-%-%'"; // Ищет все заказы
String date = today.format(DateTimeFormatter.ofPattern("dd/MM/yy")); // С косыми чертами
String orderId = String.format("ZAK-%s-%04d", date, newNumber); // ZAK-ДД/ММ/ГГ-0001
```

#### 2. Явная проверка дубликатов (SQLiteDatabase.java)

**Проблема**: При одновременном создании заказов могла возникнуть ситуация, когда два заказа получат одинаковый ID.

**Решение**: Добавлена явная проверка существования сгенерированного ID перед сохранением заказа.

**Изменения**:
```java
// Явная проверка дубликатов (хотя PRIMARY KEY уже защищает)
String checkSql = "SELECT COUNT(*) FROM orders WHERE id = ?";
try (PreparedStatement checkStmt = conn.prepareStatement(checkSql)) {
    checkStmt.setString(1, orderId);
    try (ResultSet rs = checkStmt.executeQuery()) {
        if (rs.next() && rs.getInt(1) > 0) {
            logger.warn("Сгенерированный ID {} уже существует, пробуем снова", orderId);
            orderId = null; // Повторная генерация
        }
    }
}
```

Максимум 5 попыток генерации уникального ID.

#### 3. Улучшение парсинга даты-времени для сортировки (OrderView.java)

**Проблема**: Метод `parseAppointmentDateTime()` мог некорректно работать с некорректными данными.

**Решение**:
- Добавлена валидация формата даты (должен быть dd/MM/yyyy с двумя слешами)
- Добавлена валидация формата времени (минимум 4 символа)
- Добавлена проверка корректности значений часа (0-23) и минуты (0-59)
- Улучшено логирование ошибок

**Изменения**:

### Итог

**Основное изменение**: Генерация ID теперь использует **сквозную нумерацию** (максимальный порядковый номер + 1 среди ВСЕХ заказов), что исключает возможность создания заказов с одинаковым порядковым номером и разными датами.

**Пример**:
- Если последний заказ: `ZAK-30/06/26-0010`
- Новый заказ будет: `ZAK-18/07/26-0011` (номер 0011, а не 0001)
