# Техническое задание
## Система управления автосервисом (AutoService Admin)

---

## 1. Общие сведения

### 1.1. Назначение системы
Система предназначена для автоматизации работы администратора автосервиса. Обеспечивает управление клиентами, заказами-нарядами, справочниками услуг и запчастей, а также планирование записей на ремонт.

### 1.2. Целевая аудитория
- Администраторы автосервиса
- Управляющие СТО

### 1.3. Технологии
| Компонент | Технология |
|-----------|------------|
| Язык программирования | Java 21 |
| GUI-фреймворк | JavaFX 21 |
| База данных | SQLite |
| Сборка проекта | Maven |
| Тестирование | JUnit 5, Mockito |

---

## 2. Функциональные требования

### 2.1. Дашборд администратора

**Назначение:** Сводная информация о текущем состоянии бизнеса.

**Элементы интерфейса:**
| Элемент | Описание |
|---------|----------|
| Карточка «Активные заказы» | Количество заказов в статусах: НОВЫЙ, ДИАГНОСТИКА, В РАБОТЕ, ОЖИДАНИЕ ЗАПЧАСТЕЙ |
| Карточка «Всего клиентов» | Общее количество клиентов в базе |
| Карточка «Всего заказов» | Общее количество заказов за всё время |
| Карточка «В работе (руб.)» | Сумма всех незакрытых заказов |
| Секция «Последние заказы» | Список из 5 последних заказов (ID, клиент, сумма, статус) |
| Секция «Записи на сегодня» | Записи на текущую дату (время, клиент, услуга, мастер) |
| Секция «Запчасти на остатке» | Запчасти с количеством < 10 шт. |
| Кнопка «Обновить данные» | Перезагрузка данных из БД |

### 2.2. Управление клиентами

**Назначение:** CRUD-операции с клиентами.

**Поля сущности `Client`:**
| Поле | Тип | Требования | Пример |
|------|-----|------------|--------|
| id | String | Автогенерируемый UUID | `a1b2c3d4-...` |
| name | String | Обязательно, 2-50 символов | `Иванов Иван` |
| phone | String | Обязательно, формат `+7XXXXXXXXXX` | `+79161234567` |
| carModel | String | Опционально, 2-50 символов | `Toyota Camry` |
| carNumber | String | Опционально, формат госномера РФ | `А123ВС163` |
| lastRepairDate | String | Опционально, формат `dd.MM.yyyy` | `15.01.2025` |

**Функционал:**
| Действие | Описание |
|----------|----------|
| Добавить клиента | Форма с полями: имя, телефон, модель авто, госномер |
| Редактировать клиента | Двойной клик по строке таблицы → модальное окно |
| Удалить клиента | Кнопка удаления в таблице |
| Поиск | Фильтрация по имени, телефону или госномеру |
| Таблица | Отображение: Имя, Телефон, Авто, Последний ремонт |

**Валидация:**
- Телефон: маска `+7`, 10 цифр после префикса
- Госномер: буква + 3 цифры + 2 буквы + 2-3 цифры (например, `А123ВС163`)
- Допустимые буквы в госномере: А, В, Е, К, М, Н, О, Р, С, Т, У, Х

### 2.3. Управление заказами

**Назначение:** Создание и отслеживание заказов-нарядов.

**Поля сущности `WorkOrder`:**
| Поле | Тип | Требования |
|------|-----|------------|
| id | String | Автогенерируемый UUID |
| client | Client | Обязательно, ссылка на клиента |
| services | List\<String\> | Список названий услуг |
| servicePrices | List\<Double\> | Цены услуг |
| spareParts | List\<SparePart\> | Список запчастей |
| sparePartQuantities | List\<Integer\> | Количество каждой запчасти |
| total | double | Автоматический расчёт |
| status | String | Один из допустимых статусов |

**Статусы заказа:**
| Статус | Описание |
|--------|----------|
| НОВЫЙ | Заказ создан, не начат |
| ДИАГНОСТИКА | Проводится диагностика |
| В РАБОТЕ | Ремонт выполняется |
| ОЖИДАНИЕ ЗАПЧАСТЕЙ | Ждутся запчасти |
| ГОТОВ | Заказ выполнен, готов к выдаче |
| ЗАКРЫТ | Заказ завершён, оплата получена |

**Функционал:**
| Действие | Описание |
|----------|----------|
| Создать заказ | Диалоговое окно с выбором клиента, услуг, запчастей |
| Редактировать заказ | Смена статуса, добавление/удаление услуг и запчастей |
| Удалить заказ | Физическое удаление из БД |
| Просмотр деталей | Детальная информация по заказу |
| Таблица | Отображение: ID, клиент, сумма, статус |

**Логика создания заказа:**
1. Выбор клиента из выпадающего списка
2. Выбор даты и времени записи (опционально)
3. Добавление услуг из справочника
4. Добавление запчастей с проверкой остатка на складе
5. Автоматический расчёт итоговой суммы
6. При создании записи в календаре — проверка занятости времени

### 2.4. Справочники

**Назначение:** Управление справочными данными.

#### 2.4.1. Услуги

**Поля сущности `Service`:**
| Поле | Тип | Требования |
|------|-----|------------|
| id | String | Автогенерируемый UUID |
| name | String | Обязательно, уникально, 2-100 символов |
| price | double | Обязательно, > 0 |

**Функционал:** Добавить, удалить услугу.

#### 2.4.2. Запчасти

**Поля сущности `SparePart`:**
| Поле | Тип | Требования |
|------|-----|------------|
| id | String | Автогенерируемый UUID |
| name | String | Обязательно, уникально, 2-100 символов |
| wholesalePrice | double | Опционально, >= 0 |
| retailPrice | double | Опционально, >= 0 |
| stock | int | Опционально, >= 0 |

**Функционал:**
- Добавить запчасть с ценой и остатком
- Удалить запчасть
- Внести приход (увеличение остатка) через диалог

### 2.5. Календарь записей

**Назначение:** Планирование и просмотр записей клиентов на ремонт.

**Поля сущности `Appointment`:**
| Поле | Тип | Требования |
|------|-----|------------|
| id | int | Автогенерируемый (SQLite AUTOINCREMENT) |
| client | Client | Обязательно |
| masterName | String | Обязательно, один из: Иван, Петр, Сергей, Антон |
| serviceName | String | Обязательно |
| date | String | Формат `yyyy-MM-dd` |
| time | String | Формат `HH:mm`, из списка слотов |
| orderId | String | Опционально, ссылка на заказ |

**Слоты времени:** 08:00 — 20:00 с шагом 30 минут (24 слота)

**Виды календаря:**
| Вид | Описание |
|-----|----------|
| День | Расписание по часам на выбранную дату |
| Неделя | Раскладка по дням недели с 9 колонками (время + 7 дней) |
| Месяц | Календарная сетка с краткой информацией о записях |

**Функционал:**
| Действие | Описание |
|----------|----------|
| Просмотр записей | Отображение записей выбранного периода |
| Новая запись | Диалог: клиент, мастер, услуга, дата, время |
| Редактирование записи | Изменение полей записи |
| Отмена записи | Удаление записи с подтверждением |
| Проверка занятости | Блокировка выбора занятого времени |
| Создание заказа | Опциональное создание заказа-наряда из записи |

## 3. Требования к пользовательскому опыту (UX)

### 3.1. Навигация
- Табовая навигация между основными разделами: Дашборд, Клиенты, Заказы, Справочники, Запись
- Неперекрываемые вкладки (нельзя закрыть)
- Чёткие визуальные разделители между секциями

### 3.2. Валидация ввода
- Мгновенная валидация полей ввода
- Чёткие сообщения об ошибках
- Маски ввода для телефона и госномера
- Блокировка некорректных символов в реальном времени

### 3.3. Обратная связь
- Предупреждения при попытке некорректных действий
- Подтверждения для критических операций (удаление, отмена)
- Визуальное выделение выбранных строк в таблицах
- Чередование цветов строк для читаемости

### 3.4. Производительность
- Быстрая загрузка дашборда (< 1 сек)
- Мгновенная фильтрация при поиске
- Автоматическое сохранение при закрытии приложения

### 3.5. Дизайн
- Цветовая схема: светлая тема
- Размер окна: минимум 1300×700 пикселей
- Кнопки: скруглённые углы, контрастные цвета
- Карточки статистики: тени, цветовая кодировка

## 4. Требования к данным

### 4.1. Хранение данных
- SQLite база данных
- Файл БД: `autoservice.db`
- Таблицы: `clients`, `services`, `spare_parts`, `orders`, `appointments`

### 4.2. Целостность данных
- Проверка уникальности имён услуг и запчастей
- Проверка достаточности остатка запчастей при создании заказа
- Проверка занятости времени при создании записи
- Связь записей с заказами через orderId

### 4.3. Резервное копирование
- Автоматическое сохранение при закрытии приложения
- Возможность ручного сохранения через перезагрузку данных

## 5. Ограничения

### 5.1. Технические
- Требуется Java 21
- Требуется установка JavaFX 21
- Ограничение на количество мастеров (4 фиксированных)
- Фиксированные слоты времени

### 5.2. Функциональные
- Отсутствует учёт платежей
- Отсутствует история изменений заказов
- Отсутствует многопользовательский режим
- Отсутствует экспорт данных

---

## 6. Перспективы развития

1. **Расширение ролевой модели** — добавление ролей администратор/мастер
2. **Учёт платежей** — отслеживание оплаты заказов
3. **История операций** — логирование изменений данных
4. **Экспорт отчётов** — формирование PDF/Excel отчётов
5. **Уведомления** — SMS/email напоминания клиентам
6. **Мобильная версия** — веб-интерфейс для смартфонов
7. **Расширенный каталог запчастей** — привязка к VIN, совместимость
8. **Статистика и аналитика** — графики доходов, популярности услуг
