## 🟡 P2 — `DialogsUITest` (требует обсуждения)

**Файл**: `src/test/java/com/autoservice/DialogsUITest.java`

Тестирует:
- `CreateOrderDialog`
- `EditClientDialog`
- `EditOrderDialog`
- `ImportSparePartsDialog`

### ⚠️ Главное препятствие: `stage.showAndWait()`

Все диалоги используют `stage.showAndWait()` — модально блокируют JavaFX Application Thread. Тест, запущенный на FX-потоке, зависнет. Возможные подходы:

#### Вариант A — Тестирование в отдельном потоке (без рефакторинга)
- Запускать `Dialog.show()` в отдельном `Thread`, а в тесте через `Platform.runLater` находить поля диалога и проверять/заполнять их до закрытия.
- Плюс: не трогаем продакшн-код.
- Минус: хрупкость, гонки потоков, сложно отлаживать.

#### Вариант B — Рефакторинг диалогов на `show()` + callback (вторжение в продакшн)
- Заменить `showAndWait()` на `show()` с callback по завершении.
- Плюс: тестируемость, современный паттерн.
- Минус: изменение продакшн-кода, риск регрессий, нужен обзор.

#### Вариант C — Тестировать только валидацию/модель (как сейчас)
- Оставить существующие `*DialogTest` (тестируют `Client`, `Validators` и т.д.) и не претендовать на UI-тестирование диалогов.
- Плюс: нулевой риск.
- Минус: не выполняет цель плана — реальное UI-тестирование диалогов.

**Решение**: требует обсуждения. Рекомендую Вариант B для новых диалогов и Вариант A для критичных существующих.

**Усилие**: высокое.

## 🔧 Технический долг (не из исходного плана)

### TD-1 — Оптимизация `createStage()` → `@BeforeAll`
- Сейчас `createStage()` в `@BeforeEach` создаёт новую сцену для каждого теста → медленно (отдельные тесты по 2.6s).
- Вынести в `@BeforeAll` + `@AfterAll`, переиспользовать Stage, очищать только БД.
- Ожидаемый эффект: ускорение UI-тестов в ~2 раза.
- **Усилие**: малое.

### TD-2 — Подавление шума Prism в stderr
- `BufferOverflowException` / `RejectedExecutionException` от Prism software renderer в фоновом потоке.
- На результаты не влияет, но засоряет логи CI.
- Варианты: уменьшить размер сцены, отключить pulse-таймер, фильтровать stderr.
- **Усилие**: малое.

### TD-3 — Перетегирование существующих `*DialogTest`
- `CreateOrderDialogTest`, `EditClientDialogTest`, `EditOrderDialogTest`, `ImportSparePartsDialogTest` — тег `@Tag(TestTags.UI)`, но тестируют только модели/валидацию, не UI.
- Вариант: переименовать/перетегировать в `UNIT` во избежание ложного впечатления о UI-покрытии.
- **Усилие**: малое.
